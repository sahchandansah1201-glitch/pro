# PRODUCT.md

Register: product.

## Name and purpose

**Дерматолог Про (Dermatalog Pro)** — клиническая платформа поддержки принятия решений
для дерматологов и дерматоонкологов. Продукт сопровождает врача на всём пути:
приём пациента, документирование образований, дерматоскопия, картирование тела,
поддержка ИИ/XAI по риску, заключение, отчёт пациенту, маршрутизация на повторный
визит. ИИ показывает поддержку — финальное медицинское решение принимает врач.

Цель текущей фазы: перейти от демо-режима и cloud-зависимостей к **self-hosted
полноценному продукту**, разворачиваемому на собственном сервере как единое ПО:
frontend + backend + database + file storage.

## Primary users

- **Дерматолог / дерматоонколог** — ведёт визиты, разбирает фото и дерматоскопию,
  использует ABCD и 7-point checklist, формирует заключение и отчёт пациенту.
- **Администратор клиники** — расписание, услуги, врачи, лиды, интеграции,
  аналитика клиники.
- **Системный администратор** — пользователи, роли, устройства, API-ключи,
  аудит-журналы, релизные операции (`/sys/*`).

Вторичные роли (учитываются, но не являются фокусом этой фазы): ассистент,
частный врач, пациент, оператор поддержки.

## Real workflows

1. **Пациенты** — реестр, карточка, история визитов, факторы риска, согласия.
2. **Визиты** — рабочее место визита: вкладки Imaging → Assessment → Conclusion → Report.
3. **Lesions / body map** — список образований, маркировка точек на силуэте тела,
   связывание образования с фотографиями и визитами.
4. **Imaging / assets** — загрузка обзорных фото и дерматоскопии, контроль
   качества, привязка к пациенту/визиту/образованию/точке тела, версионирование.
5. **Reports** — отчёт визита врачу и patient-safe отчёт пациенту.
6. **Audit / admin** — журнал доступа (`/sys/access-events`), пользователи,
   устройства, API-ключи, релизный статус (`/sys/release-status`).

## Demo-only today

- Роли и пользователи (`src/lib/users.ts`, `src/lib/roles.ts`) — UX-симуляция,
  переключатель ролей в шапке. Не настоящая авторизация.
- Все клинические данные (пациенты, визиты, образования, фото-метаданные,
  отчёты) — мок-данные в `src/lib/mock-data.ts` и страницах.
- Бот Telegram, мини-апп бронирования — UI-симуляция без реальной интеграции.
- Аудит-события, release-status reports, access-events — генерируются скриптами
  и моками; нет реальной БД.
- Файловое хранилище: ассеты — заглушки, нет реального object storage.
- Edge functions `api-read` / `api-write` (Supabase) — артефакты прошлой фазы,
  не считаются целевой архитектурой.

## Must become real backend behavior

- **AuthN/AuthZ**: реальная авторизация (логин/пароль, MFA опционально), сессии,
  роли в отдельной таблице (`user_roles`), серверная проверка прав, RLS-эквивалент
  на уровне backend.
- **Пациенты, визиты, образования, body map** — постоянное хранение в БД,
  CRUD через backend API.
- **Imaging / assets** — реальное object storage, метаданные в БД, контроль
  доступа по роли и принадлежности к клинике.
- **Reports** — генерация и хранение, выдача защищённой ссылкой.
- **Audit log** — серверная запись событий доступа и записи; неизменяемый журнал.
- **Admin / sys** — управление пользователями, устройствами, ключами, релизами
  через backend.
- **Device Bridge / dermatoscope** — план на отдельный desktop-компонент;
  backend принимает изображения и метаданные устройства.

## Privacy and security principles (medical data)

- Фото и клинические записи — чувствительные медицинские данные. Никаких
  реальных пациентов в репозитории, никаких секретов в frontend.
- Роли хранятся **только** в отдельной таблице ролей; никогда не на профиле/
  пользователе. Проверка прав — серверная.
- Полные dermoscopy-снимки и развёрнутые AI/XAI-данные **не уходят** во внешние
  CRM/ERP по умолчанию; во внешние системы — только защищённая ссылка и
  patient-safe резюме.
- Аудит: каждое чтение/запись чувствительных сущностей логируется (кто, когда,
  что, корреляционный id), журнал — только append.
- Согласия пациента и протоколы экспортов — обязательны.
- Patient-safe формулировки: бот и отчёты пациенту никогда не утверждают
  диагноз («рак», «меланома»). Используются формулировки «требуется осмотр
  врача», «рекомендована консультация».
- Транспорт — TLS; чувствительные ассеты — приватные бакеты, выдача по
  подписанным URL с TTL.
- Ретеншн и удаление по запросу пациента — поддерживаются на уровне модели.

## Deployment goal

Self-hosted развёртывание всего продукта на собственном сервере (или кластере)
заказчика как единая система:

- Frontend (React/Vite, статическая сборка под reverse proxy).
- Backend API (server-side приложение с авторизацией, бизнес-логикой,
  аудитом).
- Реляционная БД (PostgreSQL-совместимая) под управлением оператора.
- Object storage для медицинских ассетов (S3-совместимое или локальное
  хранилище за backend-прокси).
- Журналы, бэкапы, миграции — в составе deploy-пакета.
- Возможность изолированного контурного развёртывания (без выхода в публичный
  интернет, кроме явно сконфигурированных интеграций).

## Architectural note: Supabase

Supabase Cloud **не считается** целевой зависимостью для будущей архитектуры.
Существующие артефакты (`supabase/`, edge functions, миграции) сохраняются как
исторический контекст текущей фазы, но новые компоненты не должны добавлять
зависимость от Supabase Cloud, если это не запрошено явно.

Допустимо использовать PostgreSQL и совместимые open-source компоненты,
которые могут быть развёрнуты на собственной инфраструктуре. Решения по
конкретному backend-стэку принимаются отдельной стадией миграции.

## Architecture guardrail

Целевой путь контролируется репозиторным guardrail:

- Архитектурный контракт: `docs/architecture/stage-4a-self-hosted-product.md`.
- Локальная проверка: `npm run check:self-hosted-product`.
- Unit-проверка guardrail: `npm run test:self-hosted-product`.
- Полная проверка перед PR/release: `npm run preflight:all`.

Любая новая реальная функция для пациентов, визитов, образований, файлов,
отчётов, пользователей, ролей или аудита должна либо идти через целевой backend
API, либо оставаться явно помеченной как demo/local-only до появления backend
контракта.
