# Stage 11A-11Z · Development quality ledger

Stage 11A-11Z converts the Stage 10A-10Z prevention contract into a
repository-owned Development quality ledger for future large batches. It is a
process and evidence stage: it changes no production routes, frontend runtime,
database schema, or deploy topology.

## Batch Intake

The ledger requires every normal x2 batch to capture these facts before a
Lovable sync prompt is valid:

- current branch and base HEAD;
- included stage range;
- same Pull request justification;
- micro-PR exception status;
- expected next batch hypothesis.

Stage 11A-11Z keeps the current batch size at 26 related stages after Stage
10A-10Z. The normal path is to avoid shrinking back to small PRs unless a
hotfix or security exception is written into the batch evidence.

## Diagnostics

The diagnostics section records defects as reusable prevention work. The
critical rule is `defect_requires_prevention`: a repeated or newly discovered
defect is not considered fixed until a guard, test, command, or documented
prevention mechanism exists.

The ledger also records:

- source batch for the defect;
- concrete symptom;
- recurrence class;
- prevention rule;
- command proving the prevention;
- whether temporary artifacts were checked before handoff.

## Verification

Required verification gates are:

```bash
npm run test:stage11a-11z
npm run check:stage11a-11z
npm run ledger:stage11a-11z:dry-run
npm run preflight:stage11a-11z
npm run preflight:stage10a-10z
npm run check:project-memory
npm run typecheck
npm run preflight:all -- --dry-run
node scripts/check-no-deno-locks.mjs
git diff --check
```

Stage 11A-11Z also keeps `preflight_all_alignment` as a hard rule: new stage
preflights must be wired into `scripts/preflight-all.mjs` and covered by
`scripts/preflight-all.test.mjs`.

## Handoff

Lovable handoff remains blocked until:

1. the Pull request is merged into `main`;
2. local `main` contains the merged stage files;
3. `npm run preflight:stage11a-11z` passes on local `main`;
4. `node scripts/check-no-deno-locks.mjs` passes;
5. `package-lock.json` is unchanged.

The post-merge prompt is generated by
`scripts/stage11a-11z-development-quality-ledger.mjs` from the manifest, not
from memory. This preserves the `merge_before_lovable` rule.

## Product Boundary

- Runtime behavior changed: false.
- Backend routes added: none.
- Frontend product pages added: none.
- Database migrations added: none.
- Managed runtime/database dependency: none.
- Browser hardware APIs: false.
- External runtime calls: false.

The ledger stores repository quality metadata only. It does not contain
secrets, patient identifiers, storage references, worker raw data, or live
operator evidence.
