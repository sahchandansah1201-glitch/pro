# Stage 3H — Release reviewer FAQ for auth/assets readiness

## 1. Purpose

This FAQ helps a reviewer sign off on a release of the auth/assets
readiness scope. It is documentation only and does not change runtime
code, tests, scripts, CI, or backend configuration.

## 2. Common questions

**Q: What should I run before sign-off?**
Run the four evidence commands in section 3 on the target ref. All must
complete cleanly. Then review Stage 3A–3G docs.

**Q: Why is `M package-lock.json` expected and preserved?**
The repository intentionally keeps a preserved `M package-lock.json`
modification across these documentation stages. It must not be deleted,
manually regenerated, or reverted as part of doc-only work. Seeing it in
`git status --short` is expected.

**Q: Why is no `deno.lock` allowed?**
`deno.lock` files are generated by local `deno test` / `deno check`
runs and must never be committed. `node scripts/check-no-deno-locks.mjs`
enforces this and must pass.

**Q: Why is the real-auth smoke optional/local?**
Credential-free CI is the default. Real-auth smoke
(`npm run smoke:auth-assets`) requires doctor credentials
(`E2E_DOCTOR_EMAIL`, `E2E_DOCTOR_PASSWORD`, `E2E_VISIT_ROUTE`) which
must never be committed. It is run locally or on explicitly approved
environments only.

**Q: What must never be recorded in docs, logs, screenshots, or notes?**
- secrets
- passwords
- access tokens
- signed URLs
- storage paths
- service-role keys (`SUPABASE_SERVICE_ROLE_KEY`)

## 3. Evidence commands

```bash
npm run preflight:auth-assets
npm run test:smoke-auth-assets
node scripts/check-no-deno-locks.mjs
git status --short
```

## 4. Known expected repository state

- `node scripts/check-no-deno-locks.mjs` passes; no `deno.lock` files exist.
- `M package-lock.json` may appear in `git status --short` and is the
  known preserved modification.
- New docs changes are acceptable until committed.
- No generated artifacts, test reports, Playwright reports, or unrelated
  source/config changes should appear.
- `package-lock.json` must not be deleted, manually regenerated, or
  reverted.

## 5. Escalation

- Release owner: recorded outside repo.
- Rollback owner: recorded outside repo.
- Backend/Supabase owner: recorded outside repo.
- Security/privacy reviewer: recorded outside repo.

For live incidents follow Stage 3D. For rollback follow Stage 3B. Record
the final decision per Stage 3E.

Final documentation index lives in [docs/frontend/stage-3i-final-documentation-index.md](./stage-3i-final-documentation-index.md).
